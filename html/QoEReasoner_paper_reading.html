1. 一句话总结

QoEReasoner 不是让 LLM 直接读原始 RAN KPI 时间序列并“凭感觉诊断”，而是把 LLM 放在 planner / verifier / reporter 的位置，通过确定性工具、协议知识库和历史案例库，把 QoE 退化诊断拆成可控的 anomaly detection、fault causal chain reasoning 和 root-cause categorization 流程。

QoEReasoner = LLM planner + deterministic KPI tools + Knowledge Base constraints + Historical Bank retrieval + evidence-grounded report generation。

2. 背景与任务定义

RAN QoE 退化只是最终症状，真实原因可能隐藏在 PHY、MAC、RLC、PDCP 等多层协议 KPI 的时间动态中。论文提出系统必须满足 reliable numeric perception、causally grounded explanation 和 coherent cross-task reasoning。

输入是 session-level multivariate KPI telemetry：

$$\mathbf{X}\in\mathbb{R}^{T\times M}$$

其中 $T$ 是时间戳数量，$M$ 是 KPI 维度数。

Anomaly Detection：

$$f_{\text{AD}}:\mathbf{X}\mapsto \hat{y}^{\mathrm{deg}},\qquad \hat{y}^{\mathrm{deg}}\in\{0,1\}$$

Fault Chain Reasoning：

$$\mathbf{c}:c_1\rightarrow c_2\rightarrow\cdots\rightarrow c_L$$ $$f_{\text{FCR}}:\mathbf{X}\mapsto \hat{\mathbf{c}}=(\hat{c}_1,\ldots,\hat{c}_{\hat{L}}),\qquad \hat{c}_i\in\mathcal{C}$$

Root-Cause Categorization：

$$f_{\text{RCC}}:\mathbf{X}\mapsto \hat{y}^{\mathrm{rc}},\qquad \hat{y}^{\mathrm{rc}}\in\{1,\ldots,K\}$$

3. 系统设计

模块作用
Planner维护共享 workflow state，选择下一步诊断动作，协调 AD、FCR、RCC 和 reporter。
KPI Perception通过确定性工具把 raw KPI streams 转成 LLM 友好的 typed state objects。
FCR检索历史候选链，按 KB 逐边验证，再融合相似度和验证分数排序。
Reporter把 planner state、KPI 证据、候选链和替代假设组织成工程师可读报告。
Tool Pool预处理、统计摘要、特征选择、CNN AD classifier、RCC classifier、encoder embedding。
Knowledge Base定义 fault atom 层级、global causal graph、edge-level KPI manifestations 和专家规则。
Historical Bank保存专家验证 KPI segment 与 fault causal chain，用作候选假设初始化。

FCR 的三个阶段尤其关键：Retrieval 根据 root-cause prior 和 embedding similarity 检索 top-$n$ 候选链；Verifier 把链拆成相邻 atom pair 并查询 KB 验证 KPI 表现；Adjuster 融合 retrieval similarity 与 causal verification score，得到最终排序。

7. Case study、贡献与局限

论文展示了一个 uplink interference 诊断案例。单个 case end-to-end 诊断使用 166,375 tokens，API cost 约 0.02 美元。3 位领域专家评审 50 份报告，Correctness、Evidence Grounding、Knowledge Grounding、Interpretability 四项平均分都超过 4/5。QoEReasoner 通常约 3 分钟完成诊断，而人工专家分析通常需要 30-60 分钟。

主要贡献是把 RAN QoE 诊断从 prompt-only LLM 变成工具感知、知识约束、历史检索和状态化规划的闭环 agent。主要局限是数据集规模小且未公开，当前采用 single-dominant-chain 假设，而真实网络可能存在多条并发故障链。

实践启示：高维时间序列 + 严格领域规则 + 可解释报告这类工业诊断场景，更适合“LLM 编排 + 工具感知 + 知识约束 + 历史案例”的架构，而不是端到端 LLM 直接判断。
